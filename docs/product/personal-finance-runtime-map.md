# Карта текущего personal-finance flow

**Статус:** учебный supplement  
**Назначение документа:** зафиксировать explainable runtime map текущего shipped `personal-finance` flow end-to-end, чтобы следующие refactor slices опирались на факты, а не на guesses.  
**Связь с Техническим заданием:** supplement к `Guide-User. Спокойный review личных финансов` и learning-track issue `#133`; документ фиксирует, как текущий personal-finance сценарий реализован в коде от `App` до backend use case.  
**Источник в репозитории:** `docs/product/personal-finance-runtime-map.md`

## 1. Runtime path

| Step | Responsibility | Code anchor |
| --- | --- | --- |
| `App.tsx` | Page-level owner для `activePersonalFinanceTab`, `selectedPersonalFinanceYear`, `selectedPersonalFinanceCardId`, `personalFinanceStatus`, `personalFinanceCards`, `activePersonalFinanceSnapshots`, `selectedPersonalFinanceSettingsSnapshot` и `personalFinanceReloadTick`. Внутри `loadPersonalFinance` он загружает список карт, разрешает выбор карты, читает snapshot-ы, нормализует категории, enrich-ит карточки summary-данными и передает готовую проекцию вниз в `PersonalFinanceView`. Mutation handlers (`handleCreatePersonalFinanceCard`, `handleSaveExpenseActual`, `handleCreatePersonalFinanceTransfer`, `handleSaveIncomeActual`, `handleSaveIncomePlan`, `handleSavePersonalFinanceSettings`, `handleRenamePersonalFinanceCard`, `handleArchivePersonalFinanceCard`, `handleRestorePersonalFinanceCard`, `handleDeletePersonalFinanceCard`) тоже живут здесь. | `frontend/src/App.tsx` (`loadPersonalFinance`, `refreshPersonalFinanceDerivedViews`, `normalizePersonalFinanceSnapshotCategoryOrder`, `resolvePersonalFinanceSelection`, `resolvePersonalFinanceSettingsSelection`, `buildPersonalFinanceCardSummary`) |
| `PersonalFinanceView.tsx` | Render boundary и local UI owner. Компонент получает page-level state и callbacks сверху, после чего держит action-panel state, tab-specific UI и form-local drafts/statuses во вложенных секциях. Здесь же сосредоточены render-heavy branches и pure helpers для агрегаций/форматирования. | `frontend/src/features/personal-finance/PersonalFinanceView.tsx` (`PersonalFinanceView`, `SettingsDetails`, `CreateCardDrawerPanel`, `ExpenseEntryFormCard`, `CardTransferFormCard`, `IncomeActualFormCard`, `IncomePlanFormCard`) |
| `api/client.ts` | Typed transport facade между UI и backend API. Для чтения runtime map использует `listPersonalFinanceCards` и `getPersonalFinanceSnapshot`, а все пользовательские действия проходят через отдельные mutation methods. Эта прослойка скрывает HTTP details, но не содержит orchestration-логики. | `frontend/src/api/client.ts` (`listPersonalFinanceCards`, `getPersonalFinanceSnapshot`, `createPersonalFinanceCard`, `updateMonthlyExpenseActual`, `createPersonalFinanceTransfer`, `updateMonthlyIncomeActual`, `updateIncomePlan`, `updatePersonalFinanceSettings`, `archivePersonalFinanceCard`, `restorePersonalFinanceCard`, `deletePersonalFinanceCard`) |

Он читает `PersonalFinanceCardRepository`, `MonthlyExpenseActualRepository`, `MonthlyExpenseLimitRepository`, `MonthlyIncomeActualRepository`, `IncomeForecastRepository`, `IncomePlanRepository` и `TransactionRepository`, вычисляет limits, income statuses, `currentBalance`, `baselineAmount`, годовые totals и возвращает `Result`, который потом сериализуется controller-ом. | `backend/application/src/main/java/com/mindfulfinance/application/usecases/GetCardPersonalFinanceSnapshot.java` (`get`, `computeBalance`, `Result`) |

## 2. State owners

- `frontend/src/App.tsx` владеет page-level selection и orchestration state: какая вкладка personal-finance открыта, какой год выбран, какая карта выбрана, в каком статусе находится загрузка и какие snapshot-ы уже собраны в текущую UI projection.
- `frontend/src/App.tsx` также владеет invalidation model: `refreshPersonalFinanceDerivedViews` одновременно поднимает `personalFinanceReloadTick`, `accountsReloadTick`, `transactionsReloadTick` и `dashboardReloadTick`, то есть личные финансы здесь связаны с остальными derived views приложения.
- URL sync тоже живет в `frontend/src/App.tsx`: `syncNavigationToUrl(...)` вызывается из эффекта и использует `activePersonalFinanceTab`, `selectedPersonalFinanceYear` и `selectedPersonalFinanceCardId` как navigation source of truth.
- `frontend/src/features/personal-finance/PersonalFinanceView.tsx` владеет только local UI state, а не каноническим page state: `isActionPanelOpen` и form-local draft/submission state во вложенных формах (`cardName`, `baselineAmount`, `salaryAmount`, `bonusPercent`, `limitPercentValues`, expense draft values, transfer draft, income actual draft, income plan draft и локальные `status/errorMessage`).
- Persisted financial truth не хранится во frontend state. Frontend держит загруженную проекцию, а backend truth собирается на чтении в `GetCardPersonalFinanceSnapshot` из repository data и linked-account ledger.

## 3. Risky seams

1. `loadPersonalFinance` в `frontend/src/App.tsx` смешивает несколько обязанностей в одном effect: list fetch, selection resolution, multiple snapshot fetches через `Promise.allSettled`, category normalization, settings snapshot fallback и summary enrichment. Менять этот effect без локального safety net рискованно, потому что правила выбора карты и полноты snapshot-а распределены по нескольким helper-ам и веткам.
2. `refreshPersonalFinanceDerivedViews` связывает personal-finance с другими views сильнее, чем видно из UI. Любая personal-finance mutation не только перезагружает personal-finance, но и инвалидирует accounts, transactions и dashboard, поэтому даже локальный refactor здесь легко вызывает неожиданные побочные обновления.
3. `frontend/src/features/personal-finance/PersonalFinanceView.tsx` совмещает render tree, local form state, submit flows и pure helpers в файле на 4126 строк. Это делает seam boundaries неявными: pure calculation helpers, form logic и presentational branches находятся в одном hotspot и плохо отделяются взглядом.
4. `PersonalFinanceController.java` держит много endpoint flows и большой DTO mapping surface в одном классе. Даже если controller по intent thin, read-path и write-path share-ят validation helpers и transport mapping, поэтому расширение или extraction одного endpoint легко затрагивает соседние части.
5. `GetCardPersonalFinanceSnapshot.java` стягивает несколько repositories и сразу несколько policy/calculation concerns в одном use case: expense totals, income statuses, derived override amounts, annual and monthly limits, current balance и average values. Такой shape делает use case удобным для чтения end-to-end, но сложным для безопасного decomposition без тестовой опоры.

## 4. Retrospective

- **What became clearer:** owner page-level orchestration находится в `App.tsx`, owner local form state находится в `PersonalFinanceView.tsx`, transport seam начинается в `api/client.ts` и затем переходит в `PersonalFinanceController`, а snapshot truth собирается в `GetCardPersonalFinanceSnapshot`.
- **What became clearer:** frontend не является source of truth для personal-finance данных; он хранит проекцию, построенную из snapshot-а и повторно синхронизируемую через reload ticks.
- **What risk remains:** высокая связность между `App`, `PersonalFinanceView`, controller mapping и snapshot contract сохраняется; рефактор одного seam без safety net по-прежнему может сломать selection rules, aggregation shape или reload side effects.
- **What risk remains:** documentation slice не меняет код, поэтому hotspot-ы стали яснее, но сами coupling points остались на месте.
- **Next best seam:** `#134 Frontend: extract pure helpers from PersonalFinanceView`, потому что это самый безопасный следующий slice: сначала отделить pure calculations и formatting helpers от render/form surface, не меняя orchestration contract.
